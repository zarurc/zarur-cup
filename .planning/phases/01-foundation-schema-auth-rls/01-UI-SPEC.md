# Phase 1 — UI Design Contract

> Visual and interaction contract for the bilingual (Hebrew RTL + English LTR) Next.js 15.5 shell of Zarur-Cup / משחקי זערור. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

**Project:** Zarur-Cup / משחקי זערור (project code: ZC) — FIFA WC 2026 family prediction pool.
**Phase boundary:** Foundation shell, schema, auth, RLS. UI surface is the chrome (root layout, header, bottom tab bar, `/join`, placeholder tabs, admin shell, error pages) — feature UIs (per-match prediction, bracket pick, leaderboard) land in Phases 2 & 3.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (hand-rolled primitives) |
| Preset | not applicable |
| Component library | none — Tailwind v4.3 utilities + native HTML semantics |
| Icon library | none in Phase 1 (text-only wordmark per D-14; bottom tab bar = text labels only — see "Bottom Tab Bar" §). Heroicons reserved for Phase 2 if iconography becomes necessary. |
| Font (Hebrew) | Heebo — via `next/font/google` self-hosted, weights 400 + 700, subsets `['hebrew', 'latin']`, `display: 'swap'`, CSS variable `--font-heebo` |
| Font (Latin) | Inter — via `next/font/google` self-hosted, weights 400 + 700, subset `['latin']`, `display: 'swap'`, CSS variable `--font-inter` |

**Rationale for "Tool: none":** The locked stack (CONTEXT.md D-13..D-17 + CLAUDE.md "What NOT to Use") makes no provision for shadcn. Phase 1's UI surface is intentionally tiny — one form (`/join`), one sticky header, one bottom tab bar, four placeholder pages, and an admin shell. The discipline that actually matters (Tailwind v4.3 logical properties for RTL, Heebo/Inter locale-aware font swap, FND-03 CI lint against physical utilities) is hand-rolled at the primitives layer. Pulling in shadcn would introduce a component library that uses physical-direction utilities by default (`pl-*`, `pr-*`, `ml-*`, `mr-*`) — a direct violation of FND-03. Defer the question to v2 / Phase 6 polish if patterns become repetitive enough to warrant it.

**Locale-aware font swap (implementation contract):**

```tsx
// app/[locale]/layout.tsx
<html lang={locale} dir={dir} className={`${heebo.variable} ${inter.variable}`}>
  <body className={locale === 'he' ? 'font-heebo' : 'font-inter'}>
```

Tailwind v4 `theme` block:
```css
@theme {
  --font-heebo: var(--font-heebo);
  --font-inter: var(--font-inter);
  --font-sans: var(--font-heebo); /* default; layout overrides on /en/ */
}
```

Total font payload budget: ≤ 80 KB after `subsets` filtering and `display: 'swap'` (Heebo Hebrew+Latin 400+700 ≈ 45KB; Inter Latin 400+700 ≈ 30KB).

## Spacing Scale

8-point base scale (multiples of 4). Tailwind v4 default 0.25rem step is honored — these are the **declared tokens** the planner and executor are allowed to use; deviation requires a recorded exception below.

| Token | Value | Tailwind utility | Usage |
|-------|-------|------------------|-------|
| xs | 4px | `1` | Icon-to-label gap, inline label↔input gap inside a stacked form |
| sm | 8px | `2` | Compact element spacing, badge padding, form helper-text gap, form-field internal stack |
| md | 16px | `4` | Default element spacing, card interior padding, tab bar vertical padding |
| lg | 24px | `6` | Section padding inside `/join` hero, gap between form rows |
| xl | 32px | `8` | Layout gaps, gap between hero and form, gap between page sections |
| 2xl | 48px | `12` | Major section breaks, vertical rhythm between page hero and form |
| 3xl | 64px | `16` | Top-of-page / hero block padding (mobile: 48; ≥md: 64) |

**Exceptions (recorded):**

| Exception | Value | Why |
|-----------|-------|-----|
| Bottom tab bar height (mobile) | 56px | iOS Human Interface min thumb-zone height + safe-area inset; not on 4-step scale because the visual height is dictated by touch ergonomics, not typographic rhythm. Use `bs-14` (Tailwind 14 = 56px). |
| Touch target minimum | 44px × 44px | WCAG 2.5.5 / iOS HIG minimum. Applies to: locale toggle pill, bottom tab buttons, primary CTA. Implementation: minimum `bs-11 is-11` (44px) clickable area; visible chrome may be smaller centered inside the hit area. |
| Safe-area inset (iOS) | env(safe-area-inset-bottom) | Bottom tab bar adds `pb-[env(safe-area-inset-bottom)]` so the bar sits above the iOS home indicator. Not part of the scale; a per-device CSS env() value. |

**Generous-whitespace defaults:**
- Page section vertical padding (mobile): `pbs-12 pbe-12` (48px); (≥md): `pbs-16 pbe-16` (64px)
- Card / form-container interior padding: `pbs-6 pbe-6 ps-6 pe-6` (24px all sides), mobile-only collapse to `ps-4 pe-4` (16px) below 360px viewport width
- Page gutter (left/right viewport padding, mobile): `ps-4 pe-4` (16px); (≥md): `ps-6 pe-6` (24px)
- Between hero block and form: `mbs-8` or `mbs-12` (32-48px)

## Typography

Two weights (400 regular + 700 bold). **Four declared sizes** — collapsed to the Phase 1 cap. Hebrew Heebo and Latin Inter share the same scale tokens; only the `font-family` swaps via the locale-aware utility in §"Design System."

| Role | Size (px) | Weight | Line Height | Letter Spacing | Tailwind utility | Usage |
|------|-----------|--------|-------------|----------------|------------------|-------|
| label | 14 | 400 / 700 | 20px (1.43) | 0 | `text-sm` | Form labels, bottom tab labels, inline errors, helper text under inputs, "coming soon" sub-line on placeholder tabs, sub-wordmark, joined-at metadata, "← Back to app" link, locale-pill label |
| body | 16 | 400 | 24px (1.5) | 0 | `text-base` | Default paragraph copy, subheadings, placeholder empty-state body, input text, CTA label |
| h3 | 20 | 700 | 28px (1.4) | 0 | `text-xl` | Sticky-header wordmark, form section headings, placeholder empty-state heading, `/me` display-name h2-equivalent |
| h1 | 32 | 700 | 40px (1.25) | -0.01em on Latin, 0 on Hebrew | `text-3xl` | `/join` page heading & hero wordmark, `/admin` page heading, 404/403 headings |

**Total: 4 sizes — within Phase 1 typography cap.**

**Cap at two weights:** **400 (regular) + 700 (bold)**. Weight 500 was suggested in CONTEXT.md D-15 but is explicitly disallowed in Phase 1 — keeps the visual contract minimal and the font payload predictable. If a Phase 2 form needs a third weight, it must be re-opened in that phase's UI-SPEC.

**Hebrew-specific:**
- Heebo at body size renders ~5% larger optical x-height than Inter. **Do not** add Hebrew-only `font-size` overrides — line-height of 1.5 already accommodates both scripts. No letter-spacing on Hebrew (kerning is destructive on Hebrew letterforms).
- Bidirectional content ("2 - 1" scores, dates with month names): wrap LTR sequences in `<bdi>` or `<span dir="ltr">` per FND-06 / I18N-06. Not visible in Phase 1's UI surface but the rule is set now so Phase 2 inherits it.

**RTL spacing parity check:** Heebo and Inter at the same `text-base` `leading-normal` produce visually equivalent line heights. The shared `line-height: 1.5` body and `line-height: 1.25` h1 tokens are correct for both scripts — no per-locale tweak.

### Primary + Accent — Selected Picks with Rationale

**Primary: `#0a2540` (deep navy)** — chosen over `#0f3d2e` (forest green).
- Reasons: (1) Navy is more neutral against the 48-team flag palette — forest green collides visually with Mexico, Saudi Arabia, South Africa, Senegal kits in Phase 2 fixture cards. (2) Heebo letterforms read with sharper edges against navy than green at small sizes. (3) Forest green skews "Italian football kit" which biases visual identity toward one fanbase; navy reads as venue-neutral. (4) Cream `#fafaf7` against navy is a textbook "stadium night" pairing.

**Accent: `#f59e0b` (warm yellow / amber-500)** — chosen over `#dc2626` (red).
- Reasons: (1) Red MUST be reserved for destructive / error semantics (inline form errors, lock-breach warning in Phase 2 admin). Using red as accent would create ambiguity when Phase 2 introduces error states. (2) Yellow against navy is high-recognition stadium signage (Boca Juniors, Pacers, every sportsbook badge) — reads as "scoreboard," consistent with "modern sports clean." (3) Yellow's WCAG contrast against the cream surface is sufficient at the sizes we use it (CTAs, save indicators, focus rings — never body text).

### Accent reserved for (explicit, exhaustive list — never "all interactive elements"):

1. **Primary CTA button background** on `/join` ("Join / הצטרפו") — the single most important action in Phase 1
2. **Bottom tab bar — ACTIVE tab indicator** (a `bs-1` accent bar above the active tab's label; label itself stays primary navy for legibility)
3. **Focus-visible ring** on all interactive elements (links, buttons, inputs, tab buttons, locale pill) — `--zc-ring` = accent
4. **Saved / success indicator** copy color on the (future Phase 2) "saved" badge — Phase 1 reserves the token but does not paint it
5. **Locale toggle pill — hover state border** (subtle 2px accent border on hover, no fill; default state is muted-bordered)

Accent is NOT used for: links inside body copy (use primary navy with underline), placeholder tab "coming soon" callouts (use muted-foreground), wordmark, h1/h3.

### Destructive reserved for (explicit):

1. **Inline error text** under the `/join` form fields (wrong code, name taken, validation failure) — `text-red-600` font weight 400, 14px, with a leading `⚠` glyph (no icon library; the Unicode character is enough)
2. **`/admin/403` page heading** — for non-admins who try to reach `/admin/*` (English-only per D-05)
3. **(Reserved for Phase 2)** lock-breach integrity warning on admin dashboard (ADM-06)

### Dark Mode

**Explicitly out of scope** per REQUIREMENTS.md `POL-01 (v2 / Out of Scope)`. No dark-mode token slots reserved. The CSS variables above are declared as single values, not `:root` / `.dark` pairs. Re-opening dark mode requires a v2 UI-SPEC revision.

### A11y / Contrast

All declared text-on-surface and text-on-card pairings exceed WCAG AA (4.5:1) for normal text:
- Primary `#0a2540` on surface `#fafaf7`: **14.8:1** (AAA)
- Primary `#0a2540` on card `#ffffff`: **16.4:1** (AAA)
- Destructive `#dc2626` on surface `#fafaf7`: **5.1:1** (AA pass)
- Muted-foreground `#525252` on surface `#fafaf7`: **8.0:1** (AAA)
- Accent `#f59e0b` is **NEVER used as text color**; only as background (with primary navy text on it = 7.6:1 AAA) or as a ring/border (decorative — no contrast requirement, but the ring offsets to the surface so it remains visible)

### Error States (`/join` form, via `useActionState`)

All errors render as inline destructive-color text directly under the offending field (or under the submit button if not field-specific). Maximum one error visible at a time per field. Leading `⚠` (U+26A0) glyph, single-line where possible.

| Trigger | Hebrew | English |
|---------|--------|---------|
| Wrong invite code (`error: 'invalid_code'`) | **⚠ הקוד שגוי. בדקו שוב בוואטסאפ המשפחתי.** | **⚠ Wrong code. Check the family WhatsApp.** |
| Display name conflict (`error: 'display_name_taken'`) | **⚠ השם הזה כבר תפוס — נסו שם אחר.** | **⚠ Someone already took that name — try another.** |
| Name length out of range (`issues: min/max`) | **⚠ השם חייב להיות בין 2 ל־24 תווים.** | **⚠ Name must be 2–24 characters.** |
| Name contains disallowed characters (`issues: invalid_chars`) | **⚠ השם יכול להכיל רק אותיות עברית/אנגלית, ספרות ורווחים.** | **⚠ Name can only contain Hebrew/Latin letters, digits, and spaces.** |
| Auth / network failure (`error: 'auth_failed'` or `'profile_failed'`) | **⚠ משהו השתבש. נסו שוב בעוד רגע.** (no internal-state leak — generic) | **⚠ Something went wrong. Try again in a moment.** |
| Validation: missing required field | per-field: **⚠ שדה חובה.** | **⚠ Required.** |

**Wrong-code copy intentionally does NOT confirm "the code looks like X format" (D-03). It's a flat "wrong, check WhatsApp." Defensive against brute-force shape inference.**

### Destructive Actions in Phase 1

**None.** Phase 1 has no delete / leave-pool / merge-user UI. The `/join` form is the only mutation, and a successful join is creative, not destructive. **No destructive-confirmation copy required for Phase 1.** Destructive-confirmation copy contract is deferred to Phase 2 (admin dashboard introduces "delete fixture result," ADM-02).

### Localization Discipline

- Every user-facing string in this contract lives in `messages/{he,en}.json`. No hardcoded strings in components — even the wordmark is read from `messages` so a typo fix doesn't require a code change.
- Tab labels (next §), error glyphs, and CTA labels are message-keyed.
- DATA-04 (Hebrew team-name native-speaker review) is OUT of UI-SPEC scope but the team names will flow through `teams.name_he` / `teams.name_en` — UI-SPEC promises that the renderer respects whichever the DB delivers (per `_he` / `_en` column suffix pattern in §I18N-05).
- **Hebrew native-speaker copy review is a Phase-1 gate** for the team-name CSV (DATA-04). Reviewing the message-bundle Hebrew strings in this UI-SPEC is **Phase 2's QA-03** scope, not Phase 1's. The Hebrew strings authored above are first-pass — they will be re-validated in Phase 2.

## Token-Theming for RTL

The locale-aware font swap and the logical-property utilities together mean **there is no per-component RTL conditional** in Phase 1. The contract is:

1. **`<html dir>` is the ONLY source of direction truth** — set server-side from `params.locale` in `app/[locale]/layout.tsx`. Never mutated client-side.
2. **`<body>`'s `font-family` is the ONLY locale-aware className** — `font-heebo` on `/he/`, `font-inter` on `/en/`. All descendants inherit. Components do not pick fonts.
3. **All spacing/positioning uses Tailwind v4.3 logical-property utilities** — `ps-*`, `pe-*`, `ms-*`, `me-*`, `pbs-*`, `pbe-*`, `mbs-*`, `mbe-*`, `inset-s-*`, `inset-e-*`, `is-*`, `bs-*`, `ie-*`, `ib-*` (inline-start, block-start, etc.). The CI lint at FND-03 enforces this — `grep` rejection on any commit containing `pl-`, `pr-`, `ml-`, `mr-`, `text-left`, `text-right`, `border-l-`, `border-r-`, `left-`, `right-`.
4. **Flex direction never uses `flex-row-reverse`** — flex-row auto-flips because `<html dir>` flips the inline axis. Tested in both locales: header wordmark stays inline-start, locale pill stays inline-end, regardless of script.
5. **Bidirectional content** (LTR scores like "2 - 1" inside Hebrew copy) — wrap in `<bdi>` or `<span dir="ltr">`. Not present in Phase 1's UI surface (no fixture rows yet), but the rule is set: do NOT use Unicode LRM/RLM control characters in `messages/he.json` — they're invisible footguns.

**Verification:** Phase 1's QA gate (informal — the formal Playwright QA-01 is Phase 2) is: load `/he/` on a real iPhone in Hebrew → header wordmark is on the visual-leading side (right edge for Hebrew readers), locale pill on visual-trailing (left edge), bottom tab order shows **אני · לוח · ברקט · משחקים** read right-to-left (= Matches · Bracket · Leaderboard · Me in DOM order). Switch to `/en/` via the pill → wordmark snaps to visual-leading (left), pill to visual-trailing (right), tabs read left-to-right Matches → Me. **No visible component layout code changes between the two.**
